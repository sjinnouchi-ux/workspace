## 2. 設計思想（Codex踏襲 + Claude補足）

中核コンセプト:
> 短期戦略研究 + 小額自動執行 + 手動承認型トレード基盤

Claudeからの補足方針:

1. **執行は最後**。執行が未決定でも、記録・分析・バックテスト・AIレビューは先に完成できる。むしろ執行を急がないほうが安全で、Codexの「小さく試す」とも一致する。
2. **2段ロケット構成**にする。
   - **Stage A（記録基盤）**: 現行の07:00/22:30 cronを壊さず、実行のたびに価格・指標・シグナルをSupabaseへ「記録」する。挙動は一切変えない。
   - **Stage B（研究基盤）**: 貯まった履歴の上に、相関分析・バックテスト・月次AIレビュー・Streamlit画面を載せる。
3. **「漠然」を設計の敵にしない**。戦略が固まっていないからこそ、まず「色々な戦略を登録して比較できる箱（strategies / backtest_runs）」を先に作る。戦略は後から差し込めばよい。

## 6. 戦略フレーム（「漠然」を扱う仕組み）

戦略の中身が未確定でも前に進めるため、**戦略を"データとして登録・比較"できる箱**を先に作る。

### 戦略ステータス（Codex踏襲）

`RESEARCH`（仮説・バックテストのみ）→ `PAPER`（紙トレ検証）→ `MICRO_AUTO`（小額自動）→ `MANUAL_APPROVAL`（人間承認）
（不調なら `SUSPENDED` / `RETIRED`）

### 「メガトレンドに乗る」の候補ロジック（後日どれか選ぶ叩き台）

陣内さんが「まだ漠然」とのことなので、決め打ちせず**比較対象として複数を箱に入れる**案:

1. **長期MA転換型**: 200日線（or 50/200）の上で買い・下で退避。「上がり続ける間は乗る」古典手法。トレンドフォローの基本。
2. **モメンタム上位選択型**: 監視ユニバースの中で直近3〜6ヶ月リターン上位N銘柄に乗る。相対強度で勝ち馬に乗る。
3. **トレンド転換型**: 上昇トレンド入りで買い、下降転換で売る。大きな流れの転換点を取る。

→ 3つとも `strategies` に登録し、同じ期間・同じ手数料でバックテストして数値で比較する。「どれが良いか」は感覚でなく結果で決める。これがCodexの「厳しく振り返る」思想と一致。

## 7. 月次AIレビュー（このシステムの中核）

Codexが「最重視」と明記。Claudeも全面同意。設計の心臓に置く。

### 二段構え：先に数字、後でAI

1. **数値評価エンジン（機械判定）**: 各戦略を `PASS / WATCH / FAIL / INVALID` に自動分類
   - 指標: 勝率・期待値・平均損益・最大DD・手数料込み損益・**QQQ単純保有との比較（ベンチマーク劣後チェック）**
2. **AIの多役**: 数値判定の「理由」を説明・反証
   - 反省役（勝因敗因）/ 反証役（過剰最適化・偶然勝ちを疑う）/ リスク役（連敗・資金配分）/ 集約役（次月方針）

### 出力例（Codex案を踏襲）

```text
戦略評価: FAIL
理由: 直近30回の期待値マイナス / 勝率高いが平均損失が利益の2.4倍 /
      QQQ単純保有に劣後 / 損切り後反発が多く出口設計に欠陥
対応: 自動売買停止 → PAPER降格 → 損切り条件と保有期間を再設計
```

> 「買い候補を出すAI」ではなく「結果を冷静に裁くAI」。ここがNISA積立と違う、この趣味プロジェクトの肝。

## 9. 執行（未決定・あえて空けておく）

陣内さんと「まだ決めていない」を確認済み。設計上は**入口だけ用意**:

- 自動化レベル: Level 0(通知のみ) → 1(ペーパー) → 2(小額自動) → 3(承認付き) → 4(完全自動)
- **初期は Level 0〜1 のみ**。PaperBrokerで紙トレ検証してから判断
- 必須リスク制約（執行を入れる時に必ず）: 指値原則 / 1注文・1日上限 / 最大保有30日 / 損切り / 緊急停止 / 出金権限なしAPIキー / 発注前 `pre_trade_check`
- SBI証券は「現物ETFのAPI注文が可能か」自体が要確認。Coincheck/Binanceはread-onlyから

## 11. Codexからの引き継ぎ差分（何を変えたか）

| 項目 | Codex案 | Claude案（この設計書） | 理由 |
|---|---|---|---|
| 初期DBテーブル数 | 18個 | **6個から開始** | 使う直前に作る。管理コスト削減 |
| 着手順 | 設計→Supabase→データ→戦略→API→AI | **記録基盤を最優先で動かす** | 履歴がないと分析もAIレビューもできない |
| 執行 | 初期から設計に含める | **PaperBrokerのみ。実弾は最後** | 執行未定 + 安全最優先 |
| LINEトークン | 「再発行する」と記載 | **設計より前の最優先リスクとして独立章に** | 現在進行形で公開中のため |
| 戦略の漠然さ | 戦略前提で記述 | **戦略を箱として先に作り後で差す** | 「漠然」を設計で吸収 |
