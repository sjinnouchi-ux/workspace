# Codex→Claude: market-pilot 再設計ブリーフ

作成日: 2026-05-29  
対象プロジェクト: `market-pilot`  
目的: ETF・暗号資産の短期戦略研究、自動/手動売買判断、結果レビュー基盤の再設計

## 1. 今回の前提

このシステムは、長期保有・積立投資のための仕組みではない。

ユーザーは積立NISAを別軸で継続中であり、今回の `market-pilot` は以下を目的とする。

- 人間では監視しきれないETF・暗号資産・セクター間の動きを広く見る
- 相関関係、相対強度、トレンド、地合いを機械的に捉える
- 戦略を高速にバックテストし、感情ではなく数値で判断する
- 原則1ヶ月以内に閉じる短期・中期の売買戦略を検証する
- 小額の自動売買と、大きめの手動承認売買を分けて運用する
- 月次で結果を厳しく振り返り、次月戦略に反映する

## 2. 設計思想

### 中核コンセプト

`market-pilot` は「長期資産形成ツール」ではなく、以下の性格を持つ。

> 短期戦略研究 + 小額自動執行 + 手動承認型トレード基盤

### 重視すること

- 広く見る: ETF、セクター、債券、金、暗号資産、為替、リスク指標を監視
- 速く検証する: 戦略仮説をすぐバックテストできる
- 短く切る: 保有期間は原則1ヶ月以内
- 小さく試す: 自動売買は最小ロット・小額から始める
- 厳しく振り返る: ダメな戦略はダメと判定し、停止・降格する
- 広げられる: 将来、SBI証券、Coincheck、Binance、自動売買、AIレビューを拡張できる

## 3. 主DB方針

Google Sheetsは今回の主設計から外す。

主DBは Supabase Postgres とする。

理由:

- 型、制約、履歴管理、検索、集計に強い
- 戦略、バックテスト、注文、ポジション、レビューを正規化して扱える
- Python管理画面、バッチ処理、AIレビュー、将来のAPI実行基盤から参照しやすい

Google Sheetsは使わず、Pythonでブラウザ管理ターミナルを作る。

初期候補:

- Streamlit
- FastAPI + 簡易UI

初期構築速度を重視するならStreamlitが有力。

## 4. 全体アーキテクチャ案

```text
Data Layer
  ETF、暗号資産、為替、指標データを取得

Database Layer
  Supabase Postgresに全データを保存

Research Layer
  テクニカル、相関、相対強度、レジームを計算

Strategy Layer
  戦略ごとにシグナルを生成

Backtest Layer
  過去データで高速検証

Decision Layer
  スコアリング、資金配分、自動/手動/ペーパー分類

Execution Layer
  Paper / SBI / Coincheck / Binance / Manual Approval

Risk Layer
  1注文上限、1日上限、最大保有期間、損切り、緊急停止

Review Layer
  数値評価、AI反省、AI反証、次月戦略

Notification Layer
  LINE通知、管理画面、実行ログ
```

## 5. 売買自動化方針

初期から自動売買の入口は設計に含める。

ただし、全額自動ではなくレベル分けする。

| レベル | 内容 |
|---|---|
| Level 0 | 通知のみ |
| Level 1 | ペーパー売買 |
| Level 2 | 小額自動売買 |
| Level 3 | 自動候補 + 人間承認 |
| Level 4 | 完全自動 |

初期は Level 1〜2 を中心にする。

大きい金額の売買は手動承認とする。

### 必須リスク制約

- 成行ではなく原則指値
- 1注文あたり上限金額
- 1日あたり上限金額
- 戦略ごとの最大同時ポジション数
- 最大保有期間30日以内
- 利確、損切り、時間切れ撤退
- 発注前 `pre_trade_check`
- 緊急停止ボタン
- 注文、約定、取消、失敗をすべてSupabaseに保存

## 6. API連携方針

### SBI証券

初期から `SBIAdapter` を設計に入れる。

ただし、SBI証券で米国ETFや現物ETFのAPI注文が公式に可能かは追加確認が必要。

公式情報として確認できるのは、主に先物・オプションAPIである。  
そのため、SBI連携は最初に以下を行う。

- API利用可能範囲の確認
- 認証方式の確認
- 注文可能商品の確認
- 指値注文可否の確認
- Read-only疎通
- ドライラン発注フロー作成

### Coincheck

API連携可能。初期から暗号資産データ取得・残高照会・将来の注文に対応できる設計にする。

初期はRead-onlyから開始し、Trade権限は段階的に有効化する。

### Binance

API連携可能。Spot API、WebSocket、Testnetを想定。

日本アカウントでの利用範囲、商品制限、取引可否は実口座側で確認が必要。

初期はTestnetまたはRead-onlyを優先する。

### APIキー管理

APIキーはGitHubやSupabase DBに保存しない。

方針:

- ローカル `.env`
- OSキーチェーン
- 将来的には安全なSecrets管理

画面上では常にマスク表示。

APIキー設定は管理ターミナルから後で簡単に追加できるUIを作る。

## 8. 相関分析

相関分析は中核機能。

ダッシュボードで必ず見えるようにする。

### 必要な表示

- 相関ヒートマップ
- 相関急変アラート
- 分散リスク警告
- 注文候補の相関リスク
- 期間切り替え: 5日 / 20日 / 60日 / 120日

### 監視対象例

- セクターETF: `XLK`, `XLF`, `XLV`, `XLE`, `XLY`, `XLP`, `XLU`, `XLI`, `XLB`, `XLRE`
- テック系: `QQQ`, `VGT`, `SMH`
- リスク判定: `SPY`, `QQQ`, `HYG`, `LQD`, `TLT`, `GLD`, `UUP`
- 暗号資産: `BTC`, `ETH`, `SOL`

### 重要な考え方

単なる相関値ではなく、相関の変化を見る。

例:

- `BTC/QQQ` の相関が急低下 → 暗号資産が株式と違う動きに変化
- `XLK/XLU` の相関が急上昇 → 市場全体の一斉売買化、分散効果低下
- `GLD/QQQ` の逆相関が強化 → リスクオフ色の強まり

## 9. AIレビュー・月次振り返り

ユーザーが最も重視している内容。

買い候補を出すことより、実施後に結果を冷静に裁くことが重要。

### ターミナルに専用タブを追加

左メニューに以下を追加する。

- `AI戦略レビュー`

このタブで見る内容:

- 今月の戦略別成績
- 勝率、期待値、平均損益、最大損失、最大ドローダウン
- 手数料・スリッページ込み損益
- ペーパー売買と実売買の乖離
- ベンチマーク比較
- AI反省レポート
- AI反証レビュー
- 次月の戦略方針
- 継続 / 降格 / 停止 / 昇格判定

### AIの役割

AIは単なる感想係にしない。

最初に数字で判定し、その後AIが理由を説明・反証する。

役割分担:

- 数値評価エンジン: PASS / WATCH / FAIL / INVALID を機械判定
- AI反省役: 勝因、敗因、改善点を整理
- AI反証役: 戦略を停止すべき理由、過剰最適化、偶然勝ちを疑う
- AIリスク役: 最大損失、連敗、資金配分リスクを評価
- 集約AI: 数値と複数AI意見を統合し、次月方針を出す

### 戦略ステータス

| ステータス | 内容 |
|---|---|
| RESEARCH | 仮説段階、バックテストのみ |
| PAPER | ペーパー売買検証 |
| MICRO_AUTO | 小額自動売買 |
| MANUAL_APPROVAL | 人間承認が必要 |
| SUSPENDED | 停止 |
| RETIRED | 廃止、記録のみ保持 |

### 判定例

```text
戦略評価: FAIL

理由:
- 直近30回の期待値がマイナス
- 勝率は高いが平均損失が平均利益の2.4倍
- QQQ単純保有に劣後
- 損切り後に反発するケースが多く、出口設計に欠陥

対応:
- 現行ルールでの自動売買停止
- PAPERへ降格
- 損切り条件と保有期間を再設計
```
